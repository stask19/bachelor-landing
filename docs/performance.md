# Звіт з профілювання продуктивності (Performance Profiling)

## 1. Інструменти профілювання

Для дослідження швидкодії бекенду на базі Python/Flask було обрано наступні інструменти:

- **cProfile** (вбудований модуль C-рівня в Python) для визначення часу виконання функцій.
- **Werkzeug ProfilerMiddleware** — інтеграція cProfile безпосередньо у цикл запитів-відповідей WSGI-сервера Flask.
- **Системний Task Manager / htop** для базового моніторингу споживання оперативної пам'яті (RAM) процесом `python.exe`.

## 2. Ключові метрики продуктивності

Для інформаційної системи з модулем AHP визначено наступні метрики:

1. **Response Time (Час відповіді):** Час у мілісекундах від отримання HTTP POST запиту до відправки JSON відповіді.
2. **CPU Execution Time:** Час роботи процесора на виконання обчислень матриць NumPy.
3. **Кількість викликів функцій (Call Count):** Для виявлення зайвих циклів під час ітерації по базі автомобілів.

## 3. Тестові сценарії

- **Сценарій "Холодний старт" (Single Request):** Вимірювання часу першого запиту на розрахунок після запуску сервера (включає час на ініціалізацію модулів).
- **Сценарій "Гарячий розрахунок" (Warm Request):** Вимірювання часу повторних розрахунків матриць AHP з різними ваговими коефіцієнтами.

## 4. Результати базового профілювання

Під час тестування "Гарячого розрахунку" було зібрано статистику за допомогою `ProfilerMiddleware`. Загальний час обробки одного запиту становить близько **0.002 - 0.005 секунд**, що є відмінним показником.

### Виявлені "гарячі точки" (Hotspots):

На основі звіту профілювальника було виявлено 3 функції, які займають найбільше часу під час життєвого циклу запиту:

1. `numpy.core.multiarray.array` — ініціалізація матриць попарних порівнянь (займає основну частку математичних обчислень).
2. `numpy.core.fromnumeric.sum` — операції сумування стовпців для нормалізації матриці AHP.
3. `json.encoder.JSONEncoder.encode` — фінальна серіалізація результату з Python словника у JSON формат для відправки на клієнт.

## 5. Аналіз та висновки

Поточна продуктивність системи є дуже високою завдяки використанню оптимізованої на рівні C-коду бібліотеки NumPy. Вузьким місцем (bottleneck) є ініціалізація статичної бази автомобілів під час кожного запиту.
**Можливий шлях оптимізації:** Винесення списку автомобілів (`cars`) за межі функції маршруту або застосування мемоїзації (кешування) для повторюваних вхідних параметрів AHP-матриці.
